# Analytics Implementation Plan

## Overview

Build a client-side analytics system with cookie consent for the Verity+ blog portal. Since this is client-side only (no external analytics server), we'll create a custom lightweight tracker that stores visit data in `localStorage` and displays aggregated stats in the CMS Dashboard.

## Library Choice: Custom Lightweight Tracker

**Why custom instead of a library:** Client-side-only analytics libraries (like Plausible, Umami, Fathom) all require external servers. For a purely client-side solution with cookie consent, a custom tracker is the most appropriate — it's lightweight, fully controllable, and matches your existing architecture (Pinia + TSX + Tailwind).

**What we'll track:**

- Page views (route changes)
- Unique visitors (via anonymous session ID in localStorage)
- Time on page
- Referrer source
- Article reads (which articles are viewed)

## Implementation Steps

### Step 1: Define Analytics Types

**File:** `src/features/analytics/types.ts`

```typescript
export interface PageView {
  id: string
  path: string
  title: string
  timestamp: number
  duration: number
  referrer: string
  sessionId: string
}

export interface AnalyticsSession {
  sessionId: string
  startedAt: number
  lastActive: number
  pageViews: number
}

export interface AnalyticsSummary {
  totalVisits: number
  uniqueVisitors: number
  pageViews: number
  avgDuration: number
  topPages: { path: string; views: number }[]
  topReferrers: { referrer: string; count: number }[]
  dailyVisits: { date: string; visits: number }[]
}
```

### Step 2: Create Analytics Tracking Service

**File:** `src/features/analytics/services/tracker.service.ts`

- `getSessionId()` — generates/retrieves anonymous session ID from localStorage
- `trackPageView(path, title)` — records a page view with timestamp, referrer, duration
- `getAnalyticsSummary()` — aggregates all stored data into summary stats
- `getArticleViews()` — returns views per article slug
- `clearAllData()` — admin utility to reset analytics

Uses `localStorage` with a max entry cap (e.g., last 10,000 page views) to prevent unbounded growth.

### Step 3: Create Analytics Pinia Store

**File:** `src/features/analytics/store/analytics.store.ts`

- Reactive state for current session
- Actions: `init()`, `trackPage(path, title)`, `getSummary()`
- Computed: `isTracking`, `currentSession`
- Uses the tracker service under the hood

### Step 4: Create Cookie Consent Component

**File:** `src/features/analytics/components/CookieConsentBanner.tsx`

- Fixed bottom banner with Tailwind styling
- Multi-language support via vue-i18n
- Two buttons: "Accept" and "Decline"
- Stores consent choice in localStorage (`analytics_consent`)
- Only tracks analytics if consent === 'accepted'
- Dismisses permanently after choice is made
- Include a "Manage" link that re-shows the banner

### Step 5: Create Analytics Composable for Auto-Tracking

**File:** `src/composables/useAnalytics.ts`

- Wraps `router.afterEach()` to auto-track page views
- Checks cookie consent before tracking
- Tracks time-on-page by measuring duration between route changes
- Usage: call once in `main.ts` after router setup

### Step 6: Integrate into App

**File:** `src/main.ts`

Add after router setup:

```typescript
import { initAnalyticsTracking } from '@/composables/useAnalytics'
initAnalyticsTracking(router)
```

**File:** `src/App.tsx` or `src/features/pages/LocaleLayout.tsx`

Add `<CookieConsentBanner />` component to the public layout so it renders on all public pages.

### Step 7: Add i18n Translations

**Files:** `src/i18n/locales/en.ts`, `id.ts`, `zh.ts`
**File:** `src/i18n/types.ts`

Add `cookieConsent` section to MessageSchema:

```typescript
cookieConsent: {
  message: string
  accept: string
  decline: string
  manage: string
  learnMore: string
}
```

### Step 8: Add Analytics Section to CMS Dashboard

**File:** `src/features/cms/views/DashboardView.tsx`

Add a new card/section showing:

- Total visits (this week / this month)
- Unique visitors
- Most viewed articles (top 5)
- Simple bar chart using CSS (no external chart library needed)

### Step 9: Create CMS Analytics Detail View (Optional Future Enhancement)

**File:** `src/features/cms/views/AnalyticsView.tsx`

A dedicated analytics page accessible from CMS sidebar with:

- Date range filter
- Detailed page view breakdown
- Referrer analysis
- Export data button

## Tradeoffs & Considerations

1. **Client-side only limitation:** Data is per-browser, not aggregated across users. The CMS will only show analytics from the admin's own browser session. This is suitable for development/demo. For production, you'd need a backend endpoint to collect and aggregate data.

2. **localStorage limits:** ~5-10MB per domain. Capping at 10,000 page view entries prevents overflow.

3. **Cookie consent compliance:** The banner appears before any tracking occurs. If declined, no data is stored. This satisfies basic GDPR/ePrivacy requirements.

4. **No external dependencies:** Zero new npm packages needed. Uses existing Pinia, vue-i18n, Vue Router, and Tailwind.
